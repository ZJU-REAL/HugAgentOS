# 技能撰写指南

写 SKILL.md 正文和 description 时读本文。核心目标：写出**能被复用成千上万次**的技能，而不是
只对眼前这个例子有效的一次性脚本。

## description 是触发的关键

`description` 决定模型会不会唤起这个技能。要同时写清**做什么**和**什么时候用**——把所有"何时使用"
的信息都放进 description，别放正文。

当前模型偏向"少触发"（该用技能时没用），所以 description 要写得**主动一点**：列出典型触发语、
同义说法、相关场景，哪怕用户没明说技能名/文件类型也能命中。

- 太弱：`一个做数据看板的技能。`
- 更好：`把内部数据做成快速看板。当用户提到看板、数据可视化、内部指标，或想展示任何公司数据时都用
  本技能，哪怕没明说"看板"二字。`

## 写作风格

- **用祈使句**写指令（"先读取…"、"生成…"）。
- **解释为什么**：与其堆全大写的 MUST/NEVER，不如说清每一步的动机。今天的模型很聪明，理解了意图
  就能举一反三、超出死板指令把事办好。看到自己在写一堆 ALWAYS/NEVER 或过度死板的结构，是**黄信号**——
  回头用"解释原因"的方式重写，更有效也更耐用。
- **别过拟合**：技能要被复用无数次。别为跑通眼前这个例子塞进一堆只对它有效的琐碎规则，或写死到
  某个具体文件名/路径。遇到顽固问题，不妨换个比喻或换种工作模式，比硬加约束更可能奏效。
- **保持精简**：删掉不拉动价值的内容；如果某段指令让模型绕远路做无用功，试着删掉它看效果。

## 给输出定式

需要固定输出格式时，直接在正文里给模板：

```markdown
## 报告结构
始终用这个模板：
# [标题]
## 摘要
## 关键发现
## 建议
```

## 举例的写法

贴 1–3 个小例子往往比抽象描述更有效：

```markdown
## commit message 格式
**例 1：**
输入：给用户认证加了 JWT
输出：feat(auth): implement JWT-based authentication
```

## 无意外原则（安全红线）

技能内容**不得**含恶意代码、后门、漏洞利用，或任何可能危害系统安全的东西。一个技能的行为应与其
描述一致，不该"出人意料"。不要创建有误导性、或用于未授权访问/数据外泄等目的的技能。
（"扮演某角色"这类是可以的。）

## 迭代的思路

先写一版草稿，然后**用新眼光重看一遍**再改。真实测几个用户会说的触发语，看正文是否足以指导模型完成
任务。如果多次实跑里模型总是独立写出相似的辅助脚本，那就把它固化进 `scripts/`。改到你满意、或反馈
都没有意见为止。

> 注：Anthropic 原版 skill-creator 还带一整套"跑评测 + 基准对比 + 描述自动优化 + 浏览器评审"的机制，
> 但那些依赖 Claude Code 宿主的子智能体、`claude -p` CLI 和浏览器，**本产品的沙箱里没有**，因此本
> 技能不含那套工具——把精力放在"想清楚 + 写好 + 真跑几个例子"上即可。
