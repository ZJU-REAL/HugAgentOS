# 技能结构与渐进式披露

需要深入了解"一个技能由什么组成、怎么分层组织"时读本文。

## 目录结构

```
<技能名>/
├── SKILL.md          （必需：YAML frontmatter + Markdown 正文）
└── 可选资源
    ├── scripts/      - 确定性/重复性任务的可执行脚本（可直接跑，不必读进上下文）
    ├── references/   - 需要时才加载的参考文档
    └── assets/       - 产出里用到的文件（模板、图标、字体等）
```

## frontmatter 字段

```yaml
---
name: my-skill              # 必需。小写字母/数字/下划线/连字符，1–63 字符
description: …              # 必需。做什么 + 何时触发（最重要，见 writing-guide.md）
version: 1.0.0             # 可选
tags: [a, b]              # 可选
allowed_tools: Read, Bash   # 可选，声明技能用到的工具
---
```

- 后端技能引擎（core/agent_skills/registry.py）对 `name` 的校验是 `^[a-z0-9_-]{1,63}$`——
  **允许下划线**（与 Anthropic 原版只许连字符不同）。含空格/中文/大写的 name 会在落库时被
  自动规范化，但建议一开始就写规范。
- `description` 缺失或为空会直接落库失败。

## 渐进式披露：三层加载

技能按需分层进上下文，写的时候照这个心智模型来：

1. **元数据**（name + description）——**永远**在上下文里（约 100 词）。
2. **SKILL.md 正文**——技能触发时进上下文（**控制在 500 行内**为宜）。
3. **资源文件**（references/ scripts/ assets/）——按需读取/执行，不占常驻上下文（体量不限）。

**据此的写法：**
- 正文只写"怎么做"的主干；长文档拆进 `references/`，并在正文里清楚指路（"需要 X 时读
  `references/x.md`"）。
- 确定性的重复逻辑写成 `scripts/` 脚本，让技能**调用**而不是每次重写。
- 参考文件超过 ~300 行时，在开头放一个目录（TOC）。

## 按变体组织多领域技能

一个技能覆盖多个领域/框架时，按变体拆参考文件，正文只负责"选哪个 + 主流程"：

```
cloud-deploy/
├── SKILL.md            （主流程 + 选择逻辑）
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```

模型只会读命中的那个参考文件，省上下文。

## 什么时候加脚本

如果你发现每次用这个技能都要重复写一段相似的处理代码（比如都要写个
`build_chart.py` / `create_docx.py`），那就把它写一次放进 `scripts/`，正文里让技能直接调用——
省掉每次重新发明轮子。
