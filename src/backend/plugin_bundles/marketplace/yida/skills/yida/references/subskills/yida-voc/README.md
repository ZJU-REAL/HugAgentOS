
# VOC 信息整理

## 严格禁止 (NEVER DO)

- 不要编造日志、截图、组织 ID、应用 ID、表单 ID、时间线或复现结果。
- 不要把推测写成事实；无法确认的信息统一标记为“待补充”。
- 不要承诺平台修复时间、内部处理人或根因，除非用户提供了明确证据。
- 不要在未征得用户同意时提交外部工单或反馈；本技能默认只整理材料。

## 适用场景

**正向触发**：

- “帮我提一个 VOC/反馈/工单”
- “整理这个 bug 反馈”
- “把这个需求写成反馈”
- “这个宜搭页面很慢，帮我反馈”

## 信息采集

优先收集可验证信息：

```bash
openyida --version
node -v
openyida env --json
openyida doctor --report markdown
```

如涉及具体应用或页面，补充：

- `appType`
- `formUuid` / `processCode` / `reportId`
- 发生时间（含时区）
- 影响范围（单人、多人、整个组织）
- 复现步骤
- 期望结果
- 实际结果
- 截图、录屏、报错文本、接口响应片段（用户提供时引用）

## 写作要求

- 事实和推测分开写。
- 用户没提供的信息保留“待补充”，不要为了完整性补故事。
- 有命令输出时只摘录关键字段，不粘贴敏感 cookie、token、手机号、身份证、客户数据。
- 如果问题已能在本地复现，明确写出可复现步骤；如果不能复现，写“当前未复现，需补充环境或截图”。
