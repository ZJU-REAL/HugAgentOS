# 场景：数据看板 / 驾驶舱

> 面向决策者的「一屏看懂业务状态」。密度高、以数据为主角，图表服务于结论而非炫技。

## 场景定位

- **用户目标**：快速判断「业务健康吗 / 哪里有异常 / 趋势往哪走」。
- **不是**：门户首页（那是 workbench，偏入口）、明细管理（那是 list）。
- **典型业务**：销售驾驶舱、进销存看板、运营大屏、财务总览。

## 布局骨架 / 信息架构

先回答两个独立问题：

- **Shell（壳形态）**：页面导航骨架长什么样？单页嵌入 / 顶部导航 / 侧边导航 / L 型导航 / 隐藏导航后的自绘导航壳。
- **Archetype（页面原型）**：内容叙事怎么组织？总览 / 分析 / 监控 / 报告 / 对比 / 运营。

两者正交：单页也可以做分析型，L 型壳也可以承载总览型。不要把“看板”自动理解成一屏 KPI 平铺。

```
┌─────────────────────────────────────┐
│ 顶部：标题 + 时间范围/筛选（克制一行） │
├─────────────────────────────────────┤
│ KPI 卡组（4-6 个，有主次不平铺）       │  ← 最重要结论，一眼可见
├──────────────────┬──────────────────┤
│ 主图表（趋势/占比）│ 次图表 / 排行榜    │  ← 主大次小，非对称
├──────────────────┴──────────────────┤
│ 明细 / 排行 / 告警列表                 │
└─────────────────────────────────────┘
```

移动端：KPI 改为 2 列或横向滚动，图表纵向堆叠，保证竖屏可读。

## Shell 选型

| Shell | 业务信号 | 宜搭落地 |
|---|---|---|
| `single_page` | 单一主题 / 嵌入应用导航内 / 一屏或一页讲完 | 不自建导航，内容区直接渲染 |
| `top_nav` | 2-6 个同级专题页，层级扁平 | 导航可见时用宜搭左侧导航；导航隐藏时用 `yida-nav-shell` 顶部导航 |
| `side_nav` | 多页面后台 / 二级菜单 / 高频切换 | 优先平台导航分组；隐藏导航时用自绘侧边栏 |
| `l_shaped` | 功能模块 >=3 个，每模块 2-6 个页面，数据看板主流复杂形态 | 隐藏导航时用 `yida-nav-shell` 顶部+侧边混合；导航可见时拆成平台导航分组 |
| `no_shell` | 投屏大屏 / 分享页 / 单主题沉浸展示 | `isRenderNav=false`，只保留轻量筛选和全屏信息 |

选型优先级：PRD 明确 > 页面数量 > 功能模块层级 > 是否隐藏应用导航。拿不准时用 `single_page`，保持与宜搭应用框架融合。

## Archetype 选型

| Archetype | 适用场景 | 典型结构 |
|---|---|---|
| **总览型** | 管理层快速看全局 | KPI 行 → 趋势/构成图 → 明细/排行 |
| **分析型** | 多维下钻分析 | 全局筛选 → 分组图表 → 下钻路径 → 明细表 |
| **监控型** | 实时状态/告警 | 状态概览 → 告警列表 + 指标网格 → 小趋势 |
| **报告型** | 周报/月报/经营复盘 | 摘要结论 → 分节图表 → 行动建议 |
| **对比型** | 跨区域/渠道/时间对比 | 维度切换 → 并排图表 → 差异表 |
| **运营型** | 工作台式运营看板 | 待办/活动 Feed → 快速统计 → 重点列表 |

决策块里要写明 Shell + Archetype，例如「single_page + 总览型」或「l_shaped + 分析型」。

## 信息密度

**高**。看板要在有限空间塞下多维信息，但「高密度 ≠ 拥挤」——靠对齐、分组、留白建立秩序，别糊成一片。

## 焦点（hero）元素

**关键 KPI**。整个看板应有 1-2 个「最该先看」的核心指标，视觉上最突出（最大字号 / 置于左上视觉起点 / 点缀色强调）。

## 组件套餐（映射 design-system.md / yida-chart）

- KPI 卡：大数字（`tabular-nums`）+ 同比/环比小标 + 趋势箭头（升降用语义色）
- 图表：ECharts，走 `yida-chart` 技能；色板**克制**，对齐平台品牌变量 + 有限辅助色
- 排行/明细：`design-system.md` 列表行 / 紧凑表格
- 告警：`design-system.md` 标签（tag）+ 告警红（稀缺，只给真异常）

## Section 分组与数据叙事

- 每个页面至少 2-4 个语义分组：核心指标 / 趋势分析 / 结构分布 / 异常告警 / 明细排行。禁止 6 张以上卡片直接铺平。
- 每个图表要有一句能读懂的业务结论，落在卡片 footer、图表下方、页头 Callout 或页尾总结。
- 洞察必须包含「具体数字 + 业务含义」，例如「华东区贡献 43%，环比 +5.2pp，是本月增长主因」。禁止「数据表现良好」「趋势明显」这类空话。
- 下钻入口统一放在卡片右上角 actions 区（图标按钮），不要在卡片底部写「点击查看详情」。

## 数值格式化纪律

任何面向用户的数值都不能裸奔：

| 位置 | 要求 |
|---|---|
| KPI 主值 | 紧凑量级 + 单位，例如 `¥1,234.6万`、`12.3%` |
| 涨跌幅 | 带正负号和周期，例如 `+5.2% 环比` |
| 图表轴 | 紧凑格式，避免轴标签溢出 |
| 图表 tooltip | 完整精确格式，便于细看 |
| 表格数值列 | 右对齐 + `tabular-nums` |
| 洞察文案 | 数字 + 业务解释，不只报数 |

## 本场景专属去 AI 味要点

- ❌ **禁每张 KPI 卡配彩色渐变背景 + 一枚图标**：这是最典型的 AI 看板脸。卡片用中性底 + 数字主角，图标克制或不用。
- ❌ **禁图表色板高饱和乱撞**：每个系列都亮色 = 廉价感。用同色系深浅 + 1 个强调色区分主次。
- ❌ **禁 KPI 等大平铺**：4-6 个指标一样大 = 没有重点。核心 KPI 要更大/更突出。
- ❌ **禁裸数字**：`12345678` 直接出现在 KPI、坐标轴、tooltip、表格或洞察里都算视觉 bug。
- ❌ **禁无分组卡片瀑布**：卡片必须归入语义 section，让用户 5 秒内看懂页面有几大块。
- ✅ **图表服务于结论**：每个图表要能回答一个业务问题；纯装饰的图别放。
- ✅ **升降有语义**：增长/下降用语义色 + 方向箭头，不要只给个数字。

## 视觉方向决策块示例

```markdown
### 【视觉方向决策】
- 页面类型：dashboard（用户诉求是"进销存看板，老板一眼看库存和销售"）
- 气质关键词：专业、冷静、数据驱动
- 项目特定原则：① 库存告警（低于安全库存）必须最醒目；② 告警红只留给库存/逾期，趋势升降用绿/灰；③ 金额统一 tabular-nums 对齐
- 布局骨架：标题+时间范围 → 4 KPI(库存周转为核心，最大) → 主趋势图 + 右侧 TOP 商品 → 低库存告警表
- 信息密度：高（多维一屏，靠对齐分组建立秩序）
- 视觉焦点：库存周转 KPI + 低库存告警
- 场景专项策略：single_page + 总览型；分组为核心指标/销售趋势/商品排行/低库存告警；每张图表 footer 写一句数据洞察，金额和库存全部格式化

### 【差异化 5 维】
1. 辅助/强调色：趋势用墨绿/灰，告警用红（仅库存/逾期）；图表同色系深浅
2. 中性冷暖偏色：冷灰基底（数据专业感）
3. 圆角性格：微圆（克制，不喧宾夺主）
4. 排版性格：KPI 超大字 tabular-nums，核心 KPI 再大一号；标签用弱化文字
5. 装饰母题：KPI 左侧细色条标类别 + 非对称主次图表布局

### 【反默认说明】
KPI 用中性底 + 数字主角（非彩色渐变卡+图标），图表单色系深浅（非高饱和撞色），KPI 有主次不平铺——3 维偏离默认看板脸。

### 【图标策略】
内联 SVG，仅趋势箭头/告警用图标，KPI 卡标题不配装饰图标。
```
