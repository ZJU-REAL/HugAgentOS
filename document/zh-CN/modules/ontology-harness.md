# 领域本体治理与校验

领域本体校验为智能体增加可选的领域契约。它在工具执行前检查参数和流程，在高风险答案
生成后运行独立评审，同时保留关闭后完全不改变原对话流程的兼容路径。无论是否触发评审，
思考状态、正文增量和工具事件都会实时流式发送。

## 用户开关

登录后打开「设置 → 本体校验」。CE 实例管理员看到的入口名称为「本体治理」，同一页面
同时提供个人校验开关和全局 Domain Pack 管理。系统有已激活的默认 Domain Pack 时，可以
开启「使用领域本体校验」。

- 关闭：不注入本体、不拦截工具，使用原有流程。
- 开启但任务未命中领域工作流：不会套用无关规则。
- 开启且命中 checkpoint：首稿流式展示后，由一位独立评审员核验。
- 开启且命中 committee：多位独立委员按多数票判断通过、修订或转人工。

评审启动时，聊天右侧面板会自动展开，并单独流式展示本体校验状态与优化稿；首稿继续
保留在左侧。修订过程中的模型思考默认折叠，工具调用和新增引用仍可展开查看。用户可以
使用右上角的侧边栏按钮收起或重新展开面板，也可以通过历史消息下方的本体校验入口回看
对应结果。优化稿生成完成后，用户可以选择「替换原文」，系统不会在用户确认前覆盖首稿。

## Domain Pack

Domain Pack 是 JSON 格式的领域契约，包含四层内容：概念、关系、可执行约束和工作流。
CE 实例管理员在「设置 → 本体治理」中导入和管理版本；商业版 EE 管理员继续使用
「内容管理 → 本体治理」。

新版本导入后默认为工作草稿。每个领域包同时只能有一份工作草稿，管理员可以反复修改；
只有显式发布后，草稿才会锁定为正式版本并进入运行时。切换激活版本不修改历史审计记录。
系统随新数据库提供一个企业风险分析示例包。

工作流可以同时配置两类入口：

- `triggers`：用户问题命中文本触发词时激活；
- `asset_triggers`：工具、技能或子智能体实际被调用时，按资产 ID 或受控本体标签激活。

资产标签使用 `ontology:ConceptId` 格式。例如，把风险查询工具或风险报告技能标记为
`ontology:RiskReport`，即使用户没有说出“风险画像”等关键词，运行时实际调用该资产后也会
进入对应的本体工作流。普通展示标签不会触发本体流程。

### 预置可选标签

要让一个标签出现在技能或子智能体的选择器中，Domain Pack 必须同时声明概念和资产触发
关系。仅定义概念不会让它成为运行时触发标签。

1. 在 `concepts` 中定义概念，例如 `RiskReport`。
2. 在目标工作流的 `asset_triggers` 中选择资产类型，并把
   `ontology:RiskReport` 加入 `tags_any`。
3. 激活该 Domain Pack 版本。技能和子智能体表单随后会显示标签、关联工作流和评审级别。

以下示例让同一个标签可用于技能和子智能体：

```json
{
  "concepts": [
    {"id": "RiskReport", "name": "风险报告", "definition": "企业风险分析报告"}
  ],
  "workflows": [
    {
      "id": "enterprise_risk_analysis",
      "review_level": "committee",
      "asset_triggers": [
        {"kind": "skill", "tags_any": ["ontology:RiskReport"]},
        {"kind": "subagent", "tags_any": ["ontology:RiskReport"]}
      ]
    }
  ]
}
```

## 运行时校验

工具调用先经过不使用大模型的确定性门禁。门禁可检查 JSON Schema、必需的前置工具和
禁用工具。违规结果会连同原因和修正建议返回智能体；重复违规会要求改变策略并最终熔断。

运行时采用单调升级：一次调用可以从“未命中”升级到 checkpoint，再升级到 committee，
但不会在同一轮中降级。资产必须实际调用后才激活，系统不会仅因某个带标签的资产出现在
目录里就套用规则。已激活工作流、触发来源、工具门禁和委员会进度会作为运行事件显示在
对话中，并写入审计证据。

每轮本体治理生成唯一 `governance_run_id`。父智能体、`call_subagent` 子智能体和目标内部工具
共享同一个运行时对象；同一 `pack_id + workflow_id` 无论先由文本命中还是后由资产标签命中，
只记录一次激活。每次受治理的工具调用仍独立通过确定性门禁，因为参数和前置证据可能不同。
最终答案评审由最外层工作流唯一认领，子智能体不会自行启动委员会。子智能体内部成功完成的
工具调用会连同 `agent_id`、`sub_run_id` 和 `parent_tool_id` 归并到外层证据轨迹，并生成与普通
工具调用相同的可追溯引用。因此，领域工作流要求的工具可以由受信任的子智能体完成，不会因
外层只看到 `call_subagent` 而被误判为缺失。

高风险输出会同时检查工作流必需工具、内容长度或格式以及引用证据。委员会只在 Domain
Pack 明确配置的高风险工作流上运行，不会对每个工具调用启动大模型。

系统先运行确定性输出检查，再启动唯一一轮 checkpoint 或 committee 评审。若发现缺少必需
工具、引用或输出结构，系统会先把结构化违规信息返回给生成答案的原智能体，让它在同一个
ReAct 上下文中继续调用真实工具、搜索缺失证据并生成完整优化稿。修订指令明确要求先比较
证据缺口；如果新结论需要外部事实，必须优先补充搜索或工具调用，不能只做同义改写。

每次回答最多生成一版自动优化稿，也只执行一轮独立评审。委员会返回 `revise` 时，原智能体
仍可使用剩余的一次修订机会补充证据，但系统不会循环启动第二轮委员会。修订过程中的模型
思考单独进入默认折叠的思考区，工具调用、工具结果和优化稿正文分别流式发送，不会混入原文。

若仍需人工判断，后端会输出结构化 `manual_review` JSON，其中包含相关原句或结论、规则 ID、
风险原因和人工核对动作。前端把这些字段渲染为「领域本体人工复核」卡片，而不是把提示文本
拼接到答案末尾。用户可以继续把优化稿作为工作草稿，但不能将待人工复核内容视为已通过领域
治理的最终结论。

首稿始终按生成顺序展示在左侧；优化稿、折叠思考、补充工具调用和人工复核卡片显示在右侧
面板，并随 SSE 增量自动跟随到底部。用户主动向上滚动时，面板停止自动跟随，避免打断阅读。
优化稿不会自动覆盖首稿。用户点击「替换原文」后，系统才会原子更新消息正文，并在历史会话中
保留已采用状态。

同一开关适用于普通对话、子智能体、渠道机器人、自动化、计划、批处理和自主循环。
用户已开启本体但运行时策略不可用时，本次执行会停止并提示重试，不会静默绕过校验。

用户通过输入框显式 `@子智能体`，或者输入“调用/请调用 + 唯一完整子智能体名称 + 明确任务”
时，本轮直接进入目标子智能体，不会先让主智能体执行领域工作流再二次派发。本体运行时仍在
请求边界创建，并按子智能体 ID / 标签立即激活；随后目标子智能体内部的技能和工具调用继续
共享同一运行时，执行确定性门禁，最终输出按升级后的最高评审级别完成一次 checkpoint 或
committee 评审。

## 构建时校验

技能、MCP 工具、插件和子智能体在创建、导入、安装、上架或蒸馏保存时，会先按当前激活的
Domain Pack 检查工具依赖、禁用工具、参数 Schema 与输出契约。受控词表标签使用
`ontology:ConceptId` 格式；未知概念会在保存前被指出，普通标签不受影响。

当资产缺少工作流要求的底层工具时，校验报告会反查当前可绑定的全局 MCP，并优先显示 MCP
中文名称和工具中文名称。例如，报告会提示绑定“产业知识中心查询”，同时列出它可补齐的
“企业搜索”“企业基本信息”“企业风险预警”等能力。底层工具 ID 仍保留在完整报告中，便于
管理员排查。

新建或编辑技能时，可以在“绑定工具 (MCP)”中直接选择 MCP，不需要手工填写底层工具 ID。
后端会读取该 MCP 已发现的 `tools_json`，把工具清单用于本体构建校验。保存成功后，生成的
`SKILL.md` 会同时记录 `ontology_tags`、`ontology_workflows`、`mcp_servers` 和
`allowed_tools`。编辑、导出、上架或重新导入技能时，这些绑定关系会随技能保留。技能启用
后，其声明的 MCP 会合并到本次技能执行的能力范围，不会同时开启其他未绑定 MCP。

技能和子智能体表单把普通标签与“本体治理标签”分开。本体标签只能从当前激活领域包中
选择，并且只显示通过对应资产类型的 `asset_triggers.tags_any` 关联到工作流的标签。选择后，
表单会直接显示实际调用该资产时将触发的工作流和评审级别。

MCP 工具标签可以放在工具的 `tags` 或 `ontology_tags` 中，也可以由 MCP 服务配置统一提供。
技能使用自身元数据中的受控标签。这些标签既参加构建时检查，也可供 `asset_triggers` 在
运行时匹配。

管理员可以在 MCP 服务编辑页配置服务级标签，也可以按
`tool_name=ontology:ConceptId` 的格式配置工具级标签。用户端和管理端的技能、子智能体表单
都提供受控的本体治理标签选择器。

## 本体治理

CE 实例管理员可直接在「设置 → 本体治理」完成治理，不依赖 CE 中未提供的独立内容管理台。
商业版 EE 仍从「内容管理 → 本体治理」进入。两个入口使用同一套治理组件和服务逻辑，并提供
以下管理能力：

- 导入、校验、导出、激活和切换默认 Domain Pack；
- 查看工具门禁事件、评审证据和延迟；
- 查看门禁、评审、草案、候选合并与各来源采纳率的闭环指标；
- 从重复违规或评审修订证据自动预筛演进草案，也可手动触发；
- 把命中本体工作流后的用户点踩纠正纳入人审草案；
- 人工批准或驳回草案，并将结构化草案合并到当前工作草稿。

### 在详情页编辑版本

管理员可以打开任一领域包的「详情」，直接在总览、概念、关系、约束和工作流页签中选择
「编辑此模块」。每个页签只修改对应模块，其他模块沿用当前版本内容。

编辑器使用结构化表单，不要求管理员编写 JSON：

- 总览表单维护领域包说明、注入预算和评审阈值；
- 概念表单逐项维护定义、别名、层级、标签和受控取值；
- 关系表单从已有概念中选择起点和终点，并维护谓词、基数和禁止关系；
- 约束表单维护目标、提示和修正建议，并用字段类型、必填项、枚举、长度和数值范围配置
  JSON Schema；
- 工作流表单维护文本或资产触发条件、必需/禁止工具、输出标签和评审级别。

保存前，管理台会对组合后的完整 Domain Pack 做校验，并在表单中列出具体错误位置。首次
编辑使用管理员确认的新版本号创建一份工作草稿；后续编辑任何模块都更新同一草稿，不会为
每次保存增加版本。若领域包已有草稿，从正式版本或历史版本进入编辑时，管理台会自动切换到
该草稿继续修改，避免覆盖草稿中其他模块的变更。

工作草稿发布后会锁定为正式版本，并立即替换当前运行版本。正式版本和归档版本均为只读；
要继续修改，需从所查看的版本创建下一份工作草稿。版本管理弹窗默认每页显示 10 条，可以
切换为每页 20 条或 50 条；后台保留全部正式历史版本，不会因为分页而自动删除。管理员也可在
发布前明确放弃不再需要的工作草稿。原始 JSON 页签保留为只读核对和迁移入口。

本体演进永远不会自动修改或发布生产版本。管理员必须先审查候选，再将通过的结构化候选
合并到工作草稿，最后显式发布。会话证据和候选在入库前会经过密级拒绝、敏感信息脱敏与
注入内容净化；同一候选不能重复合并。

## API

常用接口如下。CE 设置页治理接口需要实例管理员权限；`ADMIN_TOKEN` 可作为运维回退。
商业版 EE 的 `/v1/admin/ontologies` 接口继续接受 `ADMIN_TOKEN` 或等价内容管理权限。

| 方法与路径 | 用途 |
|---|---|
| `GET/PATCH /v1/ontologies/settings` | 用户读取或更新本体开关 |
| `GET /v1/ontologies/runtime/preview` | 预览某个任务命中的规则 |
| `GET /v1/ontologies/tags?asset_kind=skill` | 读取当前可选的资产触发标签 |
| `GET /v1/ontologies/governance/access` | 探测当前 CE 用户是否可管理全局本体 |
| `/v1/ontologies/governance/*` | CE 设置页的领域包、版本、证据和演进治理 |
| `GET /v1/admin/ontologies` | 领域包与版本列表 |
| `GET /v1/admin/ontologies/tags?asset_kind=skill` | 管理端读取可选资产触发标签 |
| `GET /v1/admin/ontologies/metrics` | 闭环治理指标 |
| `POST /v1/admin/ontologies/validate` | 校验 Domain Pack JSON |
| `POST /v1/admin/ontologies/versions` | 导入新版本 |
| `PUT /v1/admin/ontologies/{pack_id}/draft` | 创建或更新唯一工作草稿 |
| `DELETE /v1/admin/ontologies/{pack_id}/draft/{version_id}` | 放弃未发布工作草稿 |
| `POST .../activate` | 发布草稿或激活指定历史版本 |
| `POST /v1/admin/ontologies/build/validate` | 预检技能、工具或子智能体资产 |
| `GET .../events`、`GET .../reviews`、`GET .../drafts` | 查看运行与演进证据 |
| `POST /v1/chats/messages/{message_id}/ontology-revision/accept` | 用本体优化稿替换原消息正文 |

## 运维提示

升级到包含本功能的版本时必须先运行数据库迁移。若用户界面显示“暂无可用本体包”，CE
实例管理员应在「设置 → 本体治理」确认至少一个包同时处于启用、默认和已有激活版本状态；
商业版 EE 管理员在内容管理台执行相同检查。

内置企业风险包更新时遵循版本不可变规则：新数据库直接激活最新内置版本；已有数据库在
没有工作草稿时自动导入一个草稿，不会覆盖当前激活版本。已有工作草稿时，本次内置更新会
推迟到草稿发布或放弃之后。管理员审查后需在「本体治理」中显式发布，资产触发规则才会
进入运行时。
